# Architecture

## How the system works

The Athena Voice Agent Tester is a Python application that uses Twilio's ConversationRelay to place outbound calls to the Pretty Good AI assessment line (+1-805-439-8008) and conduct realistic patient conversations driven by Claude. When a call is initiated via `python run.py call --scenario <id> --execute`, the system places an outbound Twilio call with a webhook URL pointing at the FastAPI server, which is exposed publicly via ngrok. Twilio answers with the scenario's opening line, then opens a persistent WebSocket connection to `/ws/conversation`. From that point forward, every time the Athena agent finishes speaking, Twilio sends a `prompt` event over the WebSocket containing the agent's transcribed speech. The server passes that text — along with the full conversation history and the scenario's goal, escalation steps, and completion signals — to Claude (claude-sonnet-4-6) via the Anthropic API. Claude responds as the patient character: short, natural, phone-appropriate turns that actively steer toward the scenario's objective. The reply is sent back over the WebSocket as a `text` event, which Twilio synthesises and plays to the agent. This loop continues until Claude signals the conversation is complete or the maximum turn count is reached. After the call, a second Claude invocation analyses the full transcript for bugs and quality issues, producing a structured markdown transcript and a per-call bug report, both saved under `outputs/`.

## Key design choices

**Claude for both sides of the loop.** Rather than scripted FSM-style patient responses, each turn is a live Claude API call with the scenario goal and full conversation history in context. This produces natural, adaptive conversation that responds appropriately to unexpected agent behaviour — much closer to a real patient than a scripted benchmark. A separate post-call Claude invocation handles transcript analysis and bug identification, keeping evaluation concerns entirely out of the real-time conversation path.

**ConversationRelay over raw TwiML.** Twilio's ConversationRelay handles ASR, TTS, barge-in detection, and turn management out of the box. This keeps the application layer clean — the server only ever deals in text — and avoids building a custom streaming audio pipeline. The `interruptible` flag is enabled so the patient bot can barge in mid-sentence, which is tested explicitly in scenario 08.

**Scenario JSON files as the single source of truth.** All scenario-specific parameters — `goal`, `opening_line`, `must_test`, `escalation_steps`, `completion_signals`, `max_turns` — live in version-controlled JSON files rather than being hardcoded. Adding a new test case requires only a new JSON file with no code changes. Claude's system prompt is assembled from the scenario at runtime, so every call is grounded in the scenario's specific objectives and knows what bugs to look for.

**Preamble filtering.** Twilio surfaces the legal recording disclaimer ("This call may be recorded for quality and training purposes.") as a prompt event before the agent speaks. A lightweight filter suppresses lines that are purely preamble while preserving any line that contains a question mark or substantive content, so the bot does not waste a turn responding to the disclaimer. Short hold phrases ("please hold", "one moment") embedded in longer substantive messages are also preserved, after discovering in early calls that the filter was incorrectly swallowing Athena's transfer notification messages.

**Thread executor for Claude calls.** The WebSocket handler runs on an async event loop. Early calls revealed that calling the Anthropic API synchronously in the handler blocked the loop for 15–23 seconds, causing Athena's silence timeout to fire and the call to drop. All Claude API calls now run via `asyncio.run_in_executor` with a `ThreadPoolExecutor`, keeping the event loop free and bringing response latency down to 2–6 seconds per turn.

**Dual-channel recording.** Every call is recorded with `recording_channels=dual` so the agent and patient voices are captured on separate tracks. Recordings are downloaded automatically via the Twilio `recording-status` callback rather than the `call-status completed` callback — an important distinction discovered in early calls where downloading immediately on call completion produced 404 errors because Twilio had not yet finished encoding the file.
